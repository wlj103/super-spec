# 5-Document Execution Protocol（五文档执行协议）

> **最终任务派发提示词参考范本。** 复制粘贴给 Agent，确保严格按规范执行不漂移。

---

请按以上5份文档执行任务，这5份文档是本次任务的完整执行规范。严格按照以下约束使用，违反即重跑：

## 一、5文档定位与权威顺序

权威链：Constitution > spec > tasks > checklist；decisions 为知识底座，歧义时回查，不参与日常执行。

- **CONSTITUTION.md** — 最高准绳，必须第一个完整读取。包含核心架构原则/已拍板决策/红线清单/术语表/占位符映射/Global Constraints。任何偏离必须先改Constitution再执行，禁止自行绕过。
- **spec.md** — 验收层：定义"成功长什么样"（What + Why + Scenario）。每个Task执行前必须回头看对应Requirement，确认自己在做什么、验收标准是什么。spec说A则tasks/checklist/decisions不能说B。
- **tasks.md** — 执行层：Task + 依赖 + 验证。按Task编号顺序推进，不得跳过、不得重排。每个Task完成后必须勾掉+附加执行证据（脚本名+通过数/线上结果），只打✅不附证据=空口通过=违规。
- **checklist.md** — 锚点层：DoD + 防走过场 + Scenario回归。全部Task完成后才逐条过checklist，不得中途部分使用。任一为否→回修重验，直到全绿。
- **decisions.md** — 知识底座：拍板决策 + 坑点 + 风险应对 + 审计方法论。执行中遇到歧义/矛盾→回查decisions→以decisions+Constitution为准。decisions变更→重跑受影响文档。

## 二、读取铁律

- 未读Constitution就执行 → 重跑。Constitution前3段（核心架构原则+已拍板决策+红线）每轮开始前必须重读，防漂移。
- 5文档必须全部完整读取，禁止只读tasks就开始干活。
- 文档中所有章节一个不能少，执行纪律的铁律必须遵守。
- 如5文档中出现"TBD"/"TODO"/"待定"/"后续补充"→文档未就绪，停止执行，回报用户补全。

## 三、执行铁律

- 主线程必须全程参与工作，不能只是委派子线程工作。
- 子线程必须和主线程相同大模型，如不能确认子线程所使用的大模型是否和主线程一致，则不能委派子进程进行代码编写及debug及优化类的重要工作，只能进行简单或重复性的辅助类工作，且工作完成后，主进程需要复查。
- 在子线程工作时，主线程也必须参加主线的任务工作，以加快进度和效率。
- 上轮有❌/未填禁止进下一轮，带伤推进=违规。
- 遇障即修，不留尾巴，"TODO:后续处理"=违规。
- 中途新决策点→停下来回报用户，不得擅自扩scope。
- 遵守红线清单——Constitution红线逐条列出，违反即回滚。
- 每完成一章，在进度追踪表中填入状态。

## 四、文档修改一致性

执行中如需修改任一文档（因新决策/发现矛盾/范围变更），必须同步修改所有受影响文档，保持5文档一致。不得只改一处不管其他。

## 五、最终验收

全部Task完成后，对照checklist校验清单逐条自查，必须实际读产物文件验证（grep/数行/检查内容），禁止不读文件直接填"是"。确认全绿再输出最终报告。

## 六、自主优化边界

除特别要求的事项，可自主优化执行方式（比如用脚本批量验证代替手动逐个curl），但不能改变步骤范围——文档要求做的必须做，不允许以"优化"为名跳过或合并。
