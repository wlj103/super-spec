**Stop vibe coding. Start spec driving.** Super-Spec 是 AI 编程 Agent 的工程纪律层——规范先行、始终验证。反偷懒、证据驱动、结构锁。为 AI 生成代码提供生产级质量门禁。专为独立开发者打造。

```
  需求文档 → G0:锚定 → G1:决策 → G2:宪法 → G3:五文档 → G4:执行 → G5:验证 → G6:收尾 → 交付
      ↓        ↓        ↓        ↓         ↓        ↓        ↓       ↓
  源文档   代码事实  决策点   最高法则   互锁规范   ≥3轮迭代  证据+   12项
  +PRD     +不变项   +理由    +红线     +任务+     全部P0   抽查    清单
          清单      +用户确认  +约束    清单        绿灯
```

## AI 写代码最大的问题

AI 编程 Agent 技术能力很强，但行为纪律很差。它们**跳过需求**、**不问就填默认值**、**零证据说"测试通过"**，交付的东西"看起来没问题"但一上线就崩。

这就是 **70% 问题**——AI 能快速帮你完成 70%，但剩下的 30% 才是生产级质量的关键。而提示词工程解决不了这个问题——模型会学会绕过提示词。"严格验证每一行代码"最终变成输出里的四个字：「验证已严格执行」。

**Super-Spec 不是提示词。它是一个结构化的强制执行系统。** 这是 AI Agent 缺少的工程纪律——规范先行、始终验证、证据驱动。

## 核心功能 — 别人没有的

Super-Spec 是唯一一个**系统性地分类模型如何偷懒**、并为每种偷懒手段创建**操作定义**来阻止的 Agent Skill。

| 功能 | 解决什么问题 | 用户会搜什么 |
|------|-------------|-------------|
| **反偷懒条款** | AI 不断跳步骤、自己填默认值、零证据说完成 | "AI总跳步骤怎么办" / "AI偷懒不按流程" / "stop AI cutting corners" |
| **文档交叉验证** | AI 代码偏离需求；规范说 A，代码做 B | "AI代码和需求不一致" / "AI不按需求开发" / "AI ignores spec" |
| **宪法驱动** | AI 多轮对话后忘记上下文；每次从零开始 | "AI忘记上下文" / "AI不记得之前的约定" / "make AI follow rules" |
| **证据验证** | AI 说"测试通过"但根本没跑；零工作证明 | "AI谎称测试通过" / "AI测试没跑就说通过" / "verify AI ran tests" |
| **单人团队级质量** | 没有 reviewer、没有 QA、没有 PM——只有你和 AI | "个人开发者用AI写代码" / "一个人用AI做项目" / "solo dev AI quality" |
| **结构锁** | AI 绕过提示词；写"已执行"但实际跳过 | "AI绕过提示词" / "提示词管不住AI" / "AI ignores instructions" |
| **抽查复验** | AI 输出表面没问题，实际有 bug；浅层验证漏掉真问题 | "AI表面没问题实际有bug" / "AI自欺欺人" / "verify AI output real" |
| **预检+回滚** | AI 部署搞崩一切，无法恢复；破坏性操作无安全网 | "AI部署搞崩了" / "AI操作失误无法恢复" / "safe AI deployment" |
| **渐进式 ≥3 轮迭代** | AI 做一轮就说"完成"，质量还很差 | "AI只做一轮就说完成" / "AI不迭代优化" / "AI doesn't iterate" |
| **七步质量流水线** | AI 编程没有结构化流程；随便写 prompt 导致随便的质量 | "AI代码如何达到生产级" / "AI工程化开发流程" / "production-grade AI code" |

## Super-Spec vs Vibe Coding

| | Vibe Coding 🎤 | Super-Spec 🛡️ |
|---|------------|------------|
| **理念** | "描述一下，AI 自己搞定" | "先写规范，始终验证，要证据" |
| **速度（第 1 小时）** | 🚀 快 | 🐢 启动稍慢 |
| **速度（第 3 天+）** | 🐌 变慢（重构、修 bug） | 🚀 稳定可预测 |
| **生产事故风险** | 高 | 低（证据锚定） |
| **理解负债** | 悄悄累积 | 零——宪法每轮锚定 |
| **最适合** | 原型、MVP、探索 | 生产环境、复杂系统、长期维护 |

## 为什么提示词工程不够用

| 做法 | 实际效果 |
|------|---------|
| "写个好 prompt" | AI 写"已遵循好 prompt"——然后继续跳步骤 |
| "加验证规则" | AI 写"验证规则已应用"——没有证据 |
| "用更长的系统提示词" | 模型忽略长提示词；上下文被稀释 |
| **Super-Spec（结构化）** | 可 grep 的清单、场景 ID、跨文档锁——无法绕过 |

Prompt 是请求。Super-Spec 是架构。**你不能靠说教让 AI 不偷懒——你得建一个它绕不过去的结构。**

## 工作原理

### 七道闸门

| 闸门 | 防止什么 | 通过条件 |
|------|---------|---------|
| **G0: 锚定审查** | 不了解当前状态就行动 | 代码事实 + 已确认决策 + 不变项清单，用户确认 |
| **G1: 决策点** | Agent 自己填默认值 | 未确认项禁止默认化；逐项询问用户 |
| **G2: 宪法** | 跨 session 理解漂移 | 七个必填段落；五文档隐式继承 |
| **G3: 五文档生成** | 偷懒、跳步、空验证 | 反偷懒条款 + 18 项验证清单 |
| **G4: 渐进式执行** | 一轮就说完成 | ≥3 轮渐进迭代；全部 P0 绿灯才继续 |
| **G5: DoD 最终验证** | 自欺欺人式验证 | 逐项证据 + 用不同方法抽查 3 个 ✅ 项 |
| **G6: 收尾** | 遗漏、残留垃圾 | 12 项收尾清单，逐项核验 |

### 五文档互锁系统

```
CONSTITUTION.md  ← 最高法则；驱动一切
      ↓
spec.md          ← 做什么 + 为什么 + 场景（验收层）
      ↔
tasks.md         ← 怎么做 + 依赖 + 证据（执行层）
      ↔
checklist.md     ← 验证 + DoD + 反敷衍（锚定层）
      ↕
decisions.md     ← 决策 + 踩坑 + 风险应对（知识库）
```

每个文档有独立职责。交叉验证捕捉漂移：spec 说 A，tasks 不能说 B。场景 ID 必须在五份文档中一致。

### 反偷懒条款

这是 Super-Spec 的知识核心。它系统性地分类了 AI 模型的**四类偷懒行为**，并为每种创建了操作定义来阻止：

| 类别 | 表现 | Super-Spec 如何阻止 |
|------|------|-------------------|
| **半结构化** | "详尽"→选择性搬运；"展开"→写 4 个字；"至少 N 个"→缩小 N | 前置计数、强制模板、操作定义 |
| **内容类型** | 模糊红线、循环定义、无数值基线、抽象错误处理 | 逐文件红线、具体定义、可测量基线、逐项错误处理 |
| **结构性** | 跳过 REMOVED 段落、无风险清单、无文件结构、场景 ID 不完整 | 强制段落、风险应对表、文件结构先于任务、场景 ID 传递 |
| **输出完整性** | 截断、AI 生成标识、不读文件说"是"、跨文档不一致 | 反截断规则、grep AI 标识、验证真实文件、跨文档一致性检查 |

## 设计原则

1. **约束行为，不约束能力** — 不指导模型已经擅长的事。锁定它偷懒的地方。
2. **结构锁，非提示词门** — 可 grep 的清单、跨文档场景 ID、不可抵赖的证据。
3. **分而治之互锁** — 五文档交叉验证。漂移可检测。
4. **宪法驱动，非记忆驱动** — 一致性来自书面文档，不靠模型记忆。
5. **探针驱动，非硬编码** — 端口、路径、依赖运行时发现。零环境耦合。
6. **证据验证，非信任验证** — "看起来没问题"不算。用不同方法抽查。

## 谁适合用

- **用 AI 编程的独立开发者** — 没有团队、没有 reviewer、没有 QA，也能获得团队级质量保证
- **被 Vibe Coding 坑过的开发者** — 快速上线，然后花几周修 AI 生成的 bug。再也不会了。
- **AI 工程师** — 想让 Agent 遵循有纪律的工程实践，带证据的那种
- **项目负责人** — 需要 AI Agent 交付可验证的、有证据支撑的产出
- **受够了 AI 偷懒的人** — "测试通过"没证据、需求被跳过、默认值悄悄填上

## 实战效果

Super-Spec 在 **V14 大脑记忆架构升级**项目中经过实战检验——一个生产级 AI 基础设施项目，46 个文件，78KB 源代码，533 个测试用例：

- **5 份文档，996 行**通过七闸门流水线生成（宪法 + spec + tasks + checklist + decisions）
- **闸门三 16 项自检**：首轮全绿，零虚假"是"回答
- **跨文档一致性**：场景 ID 在 5 份文档中全部匹配；漂移在执行前被捕获
- **零未确认默认值**：闸门一发现 8 个 Agent 会悄悄默认化的决策点
- **证据链**：每个任务完成都有脚本名 + 通过数，而非口头声称

在多个项目的测试中，Super-Spec 已证明能：
- **消除**"测试通过"零证据模式
- **防止**范围漂移，通过宪法每轮锚定
- **捕获**跨文档不一致，执行前发现
- **强制执行**渐进迭代（≥3 轮），而非一次了事
- **暴露**会被悄悄默认化的未确认决策

## 路线图

Super-Spec 解决了**工程纪律**问题——防止 AI Agent 偷懒。但方程还有另一半：

| 问题 | Skill | 状态 |
|------|-------|------|
| AI 跳步骤，零证据声称成功 | **super-spec** — 工程纪律 | ✅ 已发布 |
| AI 表达模糊，缺乏结构，留下歧义 | **super-prompt** — 提示词纪律 | 🔜 规划中 |

Super-Spec 确保 Agent **做对的事**。Super-Prompt 将确保 Agent **用对的方式表达**——精确、结构化、零歧义。一枚硬币的两面：一个约束 Agent 如何执行，一个约束 Agent 如何表达。
