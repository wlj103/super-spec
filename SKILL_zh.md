# Super-Spec

**让 AI agent 在交付前走过七道验证门——而不只是给你一个"看起来没问题"的答案。**

## 问题所在

AI 编程 agent 技术能力极强，但行为上缺乏纪律性。它们跳过步骤、不询问就填入默认值、声称"测试通过"却不提供证据、自信地交付结果却在生产环境中崩溃。传统的 prompt engineering 之所以失败，是因为模型学会了绕过 prompt——"严格执行验证"变成了输出中的四个字："验证执行严格"。

Super-Spec 不是 prompt。不是模板。它是一个**结构化执行系统**——一套完整的工程纪律，迫使 AI agent 走完每一步，产出可验证的交付物，并交给你**经过验证的、生产级的结果**。

## 有何不同

Super-Spec 是生态系统中唯一一个系统性地记录**模型如何偷工减料**并创建**可操作定义**来防止每种策略的技能。其他 spec 工具告诉 agent 要产出什么。Super-Spec 防止 agent 产出垃圾并声称完成。

| 其他 Spec 工具 | Super-Spec |
|-----------------|------------|
| "写一个 spec" | 七道 agent 必须通过的关卡；不可跳过 |
| 信任 agent 遵循指令 | 结构化锁：可 grep 的 checklist、跨文档 Scenario ID、不可否认的证据 |
| Agent 声明"完成" | Gate Five：DoD 最终验证，含抽查复验 |
| "测试通过" | 必须提供证据：脚本名 + 通过数 + 实时 e2e 结果 |
| 人工审查作为质量门 | 五文档互锁机制替代 solo 开发者的团队审查 |

## 六项设计原则

1. **约束行为，而非能力**——不要指示模型去做它已经擅长的事（架构、设计）。锁定它走捷径的环节：全覆盖、证据、双向链接。
2. **结构化锁，而非 prompt 门**——prompt 可以被绕过。只有可 grep 的 checklist、跨文档一致的 Scenario ID、不可否认的证据才能真正锁定。
3. **分而治之，互相制约**——五份文档，各有独立职责，互相交叉验证。如果 spec 写的是 A，tasks 就不能写 B。
4. **Constitution 驱动，而非记忆驱动**——一致性来源于 Constitution 文档被写下来，而非"模型记得住"。
5. **探测驱动，而非硬编码**——端口、路径、依赖、红线在运行时发现。更换环境无需触碰 skill。
6. **基于证据的验证，而非基于信任**——"看起来没问题"不计数。在最后一轮用不同方法抽查 3 个 ✅ 项。

### 模型如何绕过指令

| 你说 | 它做 |
|---------|------|
| "严格执行验证" | 写下："验证执行严格" |
| "全覆盖所有内容" | 省略 3 项，写上"其余类似" |
| "提供测试证据" | 写下："测试通过"，零证据 |
| "不要硬编码" | 硬编码路径，假装没做 |

## 七道关卡流水线

```
Requirements → G0: 锚点审查 → G1: 决策点提取 → G2: Constitution
→ G3: 五文档生成 → G4: 渐进式执行
→ G5: DoD 最终验证 → G6: 收尾 → 交付
```

| 关卡 | 防止什么 | 通过条件 |
|------|----------|----------------|
| **G0: 锚点审查** | 不了解当前状态就行动 | 代码事实 + 已确认决策 + 不变更清单，用户确认 |
| **G1: 决策点提取** | Agent 自行填入默认值 | 未确认项禁止默认填入；逐项询问用户 |
| **G2: Constitution** | 跨 session 理解漂移 | 七个必填章节已撰写；五文档隐式继承 |
| **G3: 五文档生成** | 偷工减料、跳过步骤、空验证 | 反偷工减料条款 + 18 项验证 checklist |
| **G4: 渐进式执行** | 一轮后就声称完成 | ≥3 轮渐进式执行；所有 P0 绿后才进入下一阶段 |
| **G5: DoD 最终验证** | 自欺欺人的验证 | 逐项证据 + 通过不同方法抽查 3 个 ✅ 项 |
| **G6: 收尾** | 遗漏、残留垃圾 | 12 项收尾 checklist，逐项验证 |

**不是"建议"。不完成就无法通过。**

## 五文档互锁系统

```
CONSTITUTION.md  ← 最高权威；驱动一切
      ↓ 隐式继承
spec.md          ← What + Why + Scenario（验收层）
      ↔ 双向链接
tasks.md         ← How + 依赖 + 证据（执行层）
      ↔ 双向链接
checklist.md     ← 验证 + DoD + 反敷衍（锚定层）
      ↕ 歧义回退
decisions.md     ← 决策 + 陷阱 + 风险应对（知识库）
```

每份文档有独立职责。交叉验证捕获漂移：Scenario ID 在五份文档间必须一致。基础变更触发 diff 更新。

## 触发条件

- 新项目 / 新核心模块 / 新服务 / 大版本升级
- 用户说"spec"、"纪律执行"、"别让 AI 偷工减料"
- 用户想将现有设计文档、需求文档或 prompt 文档转换为结构化、可验证的执行计划

## 前置条件

至少存在以下之一：
- 完整的设计文档（架构设计、技术选型、升级路线图）
- 需求文档（功能需求、验收标准、约束条件）
- 开发文档（API 设计、数据模型、接口规范）
- Prompt 文档（agent 行为规范、执行纪律、输出格式）

此 skill **不从零凭空创建需求**——它将现有设计文档转换为结构化的五文档系统，并交给 agent 按纪律执行。

## 铁律

违反任意一条 → 从违例点重做。

| # | 规则 | 原因（行为层面） | 违例示例 |
|---|------|------------------|-------------------|
| 1 | 生成文档前未读模板 → 重做 | 模型跳过阅读模板直接生成 | 未读 Constitution 模板就写 spec |
| 2 | 未确认的决策点禁止默认填入 → 询问用户 | 模型静默填入默认值 | 不询问就"默认使用 SQLite" |
| 3 | 禁止合并章节——每章输出独立摘要 | 模型合并以节省精力 | "Gate 三、四、五合并输出" |
| 4 | 禁止跳过章节 | 模型跳过"不重要"的章节 | 跳过 Impact 分析，直接写 Task |
| 5 | 禁止无证据通过——必须提供执行证据 | 模型说"看起来没问题" | checklist 全部"是"却无证据 |
| 6 | 每节之后：清理临时文件 → 记录变更 → 更新进度 | 模型跳过收尾步骤 | 执行完不记录；下一轮不知道改了什么 |
| 7 | 上一轮有 ❌/未填项 → 禁止进入下一轮 | 模型带伤推进 | R1 有 2 个 ❌，直接进入 R2 |
| 8 | 遇到障碍 → 立即修复，不留遗留问题 | 模型标记"TODO: 后续"来绕过 | "TODO: 在生产环境测试" |
| 9 | 遵守模板红线（动态路径、单一 token 来源、禁止静默日志） | 模型硬编码、静默失败 | 硬编码 `/home/user/` 路径 |

## Gate Zero：锚点审查

**第一步**——在讨论设计之前先理解当前状态。输出一份"锚点 Checklist"：

1. **当前代码事实**：现有工具函数、数据结构、API 参数、自动更新链——每项附位置 + 对设计的影响
2. **已确认的历史决策**：来自项目记忆、历史 spec 或用户之前的陈述
3. **明确的不变更清单**：本轮不得触碰的模块、文件或逻辑

示例：
```
当前代码事实：
- core/writer.py: write_batch() 已实现去重 + 版本链（不变更；仅调用）
- config.py: 端口/路径从 .env 读取（本轮新增 3 个环境变量）

已确认的历史决策：
- 2026-07-20: 依赖从 14 减至 10；torch 不动

明确的不变更清单：
- core/searcher.py — 本轮不变
- faiss 索引结构 — 维度不变
```

用户确认后进入 Gate One。

## Gate One：决策点提取

输出两项内容：

1. **需求理解摘要**：一句话定位 + 核心约束
2. **决策点清单**：每项含 2-4 个候选方案 + 推荐

示例：
```
D1: 依赖缩减范围
  A) 仅移除 Rerank 包（低风险，中等收益）← 推荐
  B) 用轻量替代替换 torch（高风险，大收益）
  C) 不改变依赖（零风险，零收益）
```

使用 **AskUserQuestion** 分批展示（每批最多 4 个）。

## Gate Two：生成 Constitution

所有决策确认后，生成 Constitution → `{project}/specs/{change_id}/CONSTITUTION.md`。

### 必填章节

| 章节 | 内容 | 正确示例 | 反例 |
|---------|---------|---------|----------------|
| **项目名称 + 定位** | 一句话 | "V14 大脑记忆架构升级：R2.5 依赖缩减 + 图套件" | "升级" |
| **核心架构原则** | 3-6 条不可妥协的原则 | "渐进规则：R1 扫描 → R2 审查 → R3 根因；任何代码变更后重启" | "遵循架构原则" |
| **已确认决策** | 所有 Gate One 结论 + 理由 | "D1 选择 A：仅移除 Rerank，因为 torch 是 V13 红线" | "D1 选择 A"（无理由） |
| **红线清单** | 永远不可逾越的边界 | "torch/transformers/safetensors 不动；faiss 维度不变" | "核心逻辑不变" |
| **核心术语** | 精确定义 + 跨模块映射 | "supersede = 版本化替换：旧版标记过期，新版追加；查询沿指针到最新版" | "supersede 是一种替换机制"（循环定义） |
| **模板占位符映射** | `{{PLACEHOLDER}}` → 实际值 | "`{{DB_PATH}}` → `config.get('database.path')`" | "`{{DB_PATH}}` → `/data/db`"（硬编码） |
| **全局约束** | 模型无法猜测的项目技术事实 | （见下方） | （见下方） |

### 全局约束

这些是**事实**，不是规则——版本号、路径约定、凭证获取方式。模型无法猜出它们。每个 Task 隐式继承它们。

```markdown
## 全局约束

- 语言版本：Python ≥ 3.11 / Node ≥ 18
- 数据库模式：SQLite WAL + busy_timeout / PostgreSQL 连接池限制
- 路径解析：从配置根目录解析；禁止硬编码绝对路径
- 凭证来源：环境变量优先，.env 兜底；禁止明文
- 日志标准：stdout/stderr 重定向到指定目录
- 编码：Windows 脚本 GBK+CRLF / Unix UTF-8+LF
```

### 占位符规则

1. 模板统一使用 `{{PLACEHOLDER}}`；绝不写字面值
2. 执行第一步：读取 Constitution 映射，替换所有占位符
3. 值变更只需修改 Constitution 一处；所有模板自动对齐

## Gate Three：生成五文档

```
{project}/specs/{change_id}/
├── CONSTITUTION.md    ← 最高权威
├── spec.md            ← What + Why + 验收
├── tasks.md           ← Task + 依赖 + 验证
├── checklist.md       ← DoD + 反敷衍 + Scenario 回归
└── decisions.md       ← 决策 + 陷阱 + 审计方法论
```

### 行为硬约束

**1. 生成期间禁止占位符**

spec/tasks/checklist 中禁止出现：
- "TBD" / "TODO" / "待补充" / "同上"
- "添加适当的错误处理"（未指定具体如何做）
- "为上述内容编写测试"（无内容或验收标准）
- "类似 Task N"（执行者可能乱序阅读）
- 描述要做什么却不展示怎么做（代码步骤必须包含代码/命令块）
- 引用任何 Task 中未定义的类型、函数或方法

**2. 双向链接**

spec/tasks/checklist 必须有双向引用——禁止单向引用：
```
spec.md:     "R3: 写入拒绝保护 → tasks Task 5, checklist §3 R3-1/R3-2"
tasks.md:    "Task 5: 实现写入拒绝 → spec R3, checklist §1.5"
checklist.md: "§1.5 写入拒绝验证 → spec Scenario R3-1, R3-2"
```

**3. Scenario 编号系统**

`R{需求编号}-{序号}`。checklist Part 3 必须引用每个 Scenario：
```
spec R3 有 3 个 Scenario：R3-1（正常写入）、R3-2（去重）、R3-3（拒绝）
→ checklist §3 必须包含全部 3 个及预期结果
→ tasks Task 5 必须覆盖 R3-1 到 R3-3
```

**4. 基础变更传播**

decisions.md 或 Constitution 变更 → 重新运行本 Gate → diff → 仅更新受影响章节 → 记录变更。

---

### spec.md 结构

声明式定义"成功是什么样子"。不绑定到执行路径。

**必填章节：**

- **Why**：理由 + 当前基线（含测量数据）
  ```
  正确：   "faiss_doc_count=8494, write QPS≈15/s, 14 个依赖占用 1.2GB；减至 10 个释放 400MB"
  错误：   "当前数据量大，性能需要优化"（无数值）
  ```

- **What Changes**：新增 / 修改 / 废弃 / 共享基础设施 / 文档同步 / BREAKING
  ```
  正确：
  - 新增：reranker.py（Rerank 服务封装，替代 sentence-transformers）
  - 修改：writer.py:write_batch() 增加过期标记逻辑
  - 废弃：requirements.txt 14→10 个包（移除 4 个不需要的）
  错误："优化依赖，重构写入"（无具体文件）
  ```

- **Impact**：受影响 + **明确的不变更**（红线）
  ```
  正确：   受影响：writer.py, requirements.txt, Dockerfile | 不变更：searcher.py, faiss 索引, torch
  错误：   "影响写入模块"（无文件，无不变更清单）
  ```

- **执行约束**：来自源文档的所有约束，逐条一一搬运
  - **预检计数**：生成前统计源约束（N）。生成后统计输出约束（M）。N≠M → 补充并重新运行。
  ```
  正确：   源有 5 条约束 → spec 搬运全部 5 条，编号 ①~⑤
  错误：   源有 5 条 → spec 搬运 3 条 + 写"其余类似"
  ```

- **ADDED / MODIFIED / REMOVED Requirements**：每条含 SHALL 声明 + Scenario。REMOVED 必须存在（即使为"无"），含移除原因 + 过渡方案。

- **回滚机制**：触发条件 + 目标 + 强制四列表格：
  ```
  | 步骤 | 动作 | 命令/操作 | 验证 |
  |------|--------|-------------------|--------------|
  | 1    | 停止服务 | taskkill /F /PID xxx | 所有端口关闭 |
  | 2    | 恢复数据 | xcopy .backup/xxx .data/ /E | 文件数匹配 |
  ```

**自包含规则**：核心数据结构（schema）、接口签名、配置字段和错误码——如果存在——必须在 spec 中完整写出。spec 是验收锚点。对这些内容写"参见 tasks.md" = 失败。

**格式规则**：必须包含正确/反例对：
```
格式规则：{规则描述}
正确：{具体值，真实示例}
反例：❌ {具体错误值} → ✅ {正确形式}
```

**复合约束**：必须展开——"严格执行验证"是四个字的空话：
```
正确："① 数据格式验证：检查 JSON schema 中每个必需字段存在且类型正确
        ② 一致性验证：faiss_doc_count 增量 = 本批写入记录数
        ③ 失败处理：回滚本批 + 写入 error_log + 返回 HTTP 503"
错误：  "严格执行验证"
        "错误处理"（零具体内容）
```

### 反偷工减料条款

这是 Super-Spec 的智力核心。以下关键词在 skill 中出现时，必须按其可操作定义执行。禁止缩水、省略或降级。

**A. 半结构化约束（最常被偷工减料）**

| 关键词 | 可操作定义 | 常见偷工减料 | 检查项 |
|---------|------------------------|-----------------|-------|
| **全覆盖（Exhaustive）** | 从源中搬运每一项。预检：先统计源项数（N），后统计输出项数（M）。N≠M = 失败。 | "选择性搬运"、"仅关键项"、"其余类似" | #4 |
| **展开（Expand）** | 每个锁/步骤/阶段必须指定：① 检查什么 ② 如何验证 ③ 失败怎么办。禁止只写名称。 | "严格执行验证"（4 个字）、"错误处理"（无具体内容） | #3 |
| **至少 N（At Least N）** | N 是下限。"至少 5 轮" = 最少 5 轮；不得缩水到 3 轮。 | "至少 5 轮" → 写 3 轮；"至少 1 个异常测试" → 写 0 个 | #11 |
| **强制模板（Mandatory Template）** | 模板结构必须完整填写。禁止只写模板名称。 | "使用四列表格" → 写纯文本；"附示例" → 写"正确/错误格式" | #2/#5 |
| **列出所有项（List All Items）** | 源有 N 项 → 输出必须包含全部 N 项，编号可数。 | 写 ①②③，省略 ④；多项合并为一项 | #3/#4 |

**B. 内容类型约束（模型用抽象替代具体）**

| 场景 | 可操作定义 | 常见偷工减料 | 检查项 |
|----------|------------------------|-----------------|-------|
| **红线清单** | 按文件/模块。禁止模糊的"不动现有代码"。 | "现有服务不动"、"核心逻辑不变" | #12 |
| **术语** | 每条术语 = 一句话定义 + 跨模块映射。禁止循环定义。 | "supersede 是一种版本化替换机制"（无旧版本处理细节） | #12 |
| **基线测量** | Why 章节基线数据必须包含具体数值。禁止"N 条记录"、"基本正常"。 | "数据量大"、"性能可接受" | #12 |
| **接口/配置** | Schema 必须完整写出。接口签名、配置字段、错误码——如果存在，必须在 spec 中。存在但未写 = 失败。 | 有 schema，缺接口/配置/错误码 | #1/#13 |
| **具体 > 抽象** | "验证覆盖" → 逐条列出"验证 ①…②…③…"。"错误处理" → 指定什么错误 + 如何处理。 | "Lint 验证"、"错误处理"、"适当配置" | #3 |

**C. 结构约束（模型会跳过/遗漏）**

| 场景 | 可操作定义 | 常见偷工减料 | 检查项 |
|----------|------------------------|-----------------|-------|
| **REMOVED 章节** | 必须存在（即使为"无"）。每项：移除 + 原因 + 过渡方案。 | 完全跳过 REMOVED | #14 |
| **风险应对** | decisions.md 必须包含风险列表：风险 + 严重性 + 触发阶段 + 应对 + 验收标准。 | 不列出，或"参见 spec" | #15 |
| **执行纪律** | tasks 头部必须从 Constitution 原则中逐条提取。缺少任何一条 = 失败。 | 写 2-3 条，或"遵循架构原则" | #14 |
| **文件结构** | tasks 必须在 Tasks 之前列出文件结构。每个文件标注 Create/Modify/Test。 | 跳过直接写 Tasks | #14 |
| **Scenario 搬运** | checklist Part 3 必须搬运每个 Scenario + 预期结果。仅 ID = 失败。 | "R1-1~R1-5"（无预期结果） | #9 |

**D. 输出完整性（模型会截断/留痕迹）**

| 场景 | 可操作定义 | 常见偷工减料 | 检查项 |
|----------|------------------------|-----------------|-------|
| **防截断** | 若文件预计 >8000 字，P0 尾部内容（约束/风险/回滚）必须前移。截断 = 生成失败。 | 尾部风险表/REMOVED 被截断 | #16 |
| **移除 AIGC 元数据** | 输出不得包含 AIGC 元数据或"AI 生成"页脚。grep 每个文件并移除。 | 保留平台注入的 AIGC 标记 | #16 |
| **不自欺的验证** | checklist 必须对照实际输出文件验证（grep、行数、内容检查）。禁止未读就填"是"。 | 一秒内填了 10 个"是" | #16 |
| **跨文档一致性** | 如果 spec 写 A，tasks/checklist/decisions 不能写 B。Scenario ID 必须一致。 | spec R1 有 5 个 Scenario；checklist 只列 3 个 | #9 |

### tasks.md 结构

**头部：执行纪律**——从 Constitution 核心原则中提取。禁止模糊的"遵循架构原则"。

```
## 执行纪律
1. 渐进规则：R1 扫描 → R2 审查 → R3 根因 → R4 验证 → R5 终审
2. 代码变更后重启：任何 .py/.json 变更必须重启服务后才能验证
3. 验证 ≠ 执行：运行测试 ≠ 功能正确；必须检查日志 + 输出
4. 主线程纪律：写操作仅限主线程；禁止并发 faiss 写入
```

**Task 分解**——每个 Task 包含子任务 + 可判定的验证标准。完成后必须附上：
- **证据**：脚本名 + 通过数 / 实时 e2e 结果（不可否认）
- **关键修复**：遇到的陷阱（可选）
- **偏差记录**：目标 vs 实测 + 接受理由（可选）

**Task 分解铁律：**
- **破坏性操作 → 预检**：第一个子任务必须是预检：
  ```
  - [ ] {N}.0: 预检（{操作}前）：{检查1} + {检查2} + ... ；条件不满足则停止
  ```
- **P0 风险 → 兜底保护**：必须有兜底子任务：
  ```
  - [ ] {N}.x: {风险} 兜底：{正常路径} 失败 → {兜底动作}（{为何不能阻塞上游}）
  ```
- **切换 → 回滚演练**：必须包含回滚演练子任务：
  ```
  - [ ] {N}.x: 回滚演练：按 spec 回滚表逐步执行（dry 模式）；验证每步通过
  ```

**尾部**：依赖图。

### checklist.md 结构

**Part 1：DoD 量化验证**——实例模板 DoD + 模块特定项。每项："是/否 + 证据"

**Part 2：反敷衍自检**

| # | 陷阱 | 防御 |
|---|------|---------|
| 1 | 验证 ≠ 执行 | 含"验证/确认/测试"的验收项必须有执行证据 |
| 2 | 修复 ≠ 已验证修复 | 修复后，重启 + 重新运行该节的验收 |
| 3 | 全绿 ≠ 真绿 | 最后一轮：用不同方法抽查 3 个 ✅ 项 |
| 4 | 上下文断裂 | 每轮开始：列出前一轮关键问题；每阶段：重新阅读 Constitution |
| 5 | 快乐路径偏见 | 每节至少 1 个异常输入测试 |
| 6 | 修复引入新问题 | 修改 shared/ 或接口后，立即重新运行集成测试 |
| 7 | 兜底路径敷衍 | 每个兜底/保护机制必须有 ≥1 个触发验证 |

## Gate Four：渐进式执行

用户批准后：
1. 按 tasks.md 推进。每节之后：清理临时文件 → 记录变更 → 更新进度
2. 每节/阶段输出独立摘要（≥3 行）
3. **每轮开始前重新阅读 Constitution 前 3 个章节**（原则 + 决策 + 红线），防止漂移
4. 完成每个 Task 后，勾选并附上证据 + 修复 + 偏差
5. 中途出现新决策点 → 暂停并回到 Gate One；禁止未经授权扩大范围
6. 所有 P0 绿后进入下一轮

## Solo 开发者优化

传统 spec 框架假设团队——代码审查、PR 流程、有人兜底。Solo 开发者 + AI 完全没有这些。一个错误就是一次生产事故。

| 团队假设 | Solo + AI 现实 | Super-Spec 应对 |
|-----------------|-------------------|---------------------|
| 执行者仔细阅读 spec | AI"智能"绕过规则 | 4 类反偷工减料 + 18 项 checklist |
| 有人审查质量 | 一个人 + AI | 五文档互锁替代团队审查 |
| 记忆/上下文可靠 | 长对话漂移；跨 session 丢失 | 每轮重读 Constitution；不依赖记忆 |
| 环境稳定 | 频繁换机/端口/依赖变更 | 探测驱动；零硬编码 |
| "看起来没问题"能通过 | 无人兜底 | 基于证据 + 抽查复验 |
| Spec 被参考 | Spec 被遗忘 | Gate Three → Five → Six 持续锚定 spec |

## 参考模式

以下是来自真实项目的验证实践。非强制——值得了解的经过验证的方法。

### 渐进节奏

| 轮次 | 节奏 | 意图 |
|-------|--------|--------|
| R1 | 扫描——找出所有问题；修复 P0 阻塞 | 建立问题全景 |
| R2 | 审查——R1 修复是否引入新问题 + 遗漏边界 | 防止修复引入 bug |
| R3 | 根因——检查同类问题是否存在于别处；系统性修复 | 系统性，非打地鼠 |
| R4 | 验证——前三轮是否引入新 bug | 二阶验证 |
| R5 | 终审——用不同方法抽查 3 个 ✅ 项 | 反敷衍 |

小项目：≥3 轮。大项目：≥5 轮，可能 7-10+。

### 阶段划分

1. **环境检查 + 基线** → 基线数据完整可复现
2. **逐服务审计** → P0 数量确认无遗漏
3. **集成验证** → 核心链路无断点
4. **集中修复** → 问题清零 + 触发式预防
5. **边界 + 安全** → 所有边界通过 + 无高危安全问题
6. **清理 + 压力测试 + 收尾** → 性能达标 + 无泄漏 + 文档一致

小项目可将 2-4 合并为一轮"审计 + 修复 + 验证"。

### Task 分解示例

好的——每个 Task 遵循文件边界；2-5 分钟可判定：
```markdown
- [ ] Task 3: 实现 data_writer.py（依赖 Task 1）
  - [ ] 3.1: write_record() 基础写入 + 去重检查
  - [ ] 3.2: 版本链追加 + 过期标记
  - [ ] 3.3: 输入验证 + 拒绝非法数据
  - 验证：test_data_writer.py 42/42 PASS
```

差的——太模糊：
```markdown
- [ ] Task 3: 完成所有 data_writer 功能
  - 验证：测试通过
```

### 文件结构

在写 Tasks 之前锁定文件边界：
```markdown
## 文件结构
- Create: `service/matcher.py` — 匹配调度逻辑
- Create: `service/calibrator.py` — 校准学习器
- Modify: `core/server.py:845-920` — 写入网关 → data_writer
- Modify: `requirements.txt` — 12→8（依赖缩减）
- Test: `tests/test_data_writer.py` — CRUD + 版本链 + 验证
```

### 证据附上

```markdown
- [x] SubTask 3.1: write_record() 基础写入 + 去重
  - 证据：test_data_writer.py 42/42 PASS；实时 e2e ALL PASS
  - 关键修复：① record_id 键名兼容修复 ② 并发写入路径隔离
  - 偏差记录：目标 ≤200ms，实测 ~350ms；按 ≥10x 退化规则接受
```

### Scenario 覆盖维度

每个 Requirement 按需选择：快乐路径 / 兜底失败 / 边界值 / 并发安全 / 鉴权 / 数据迁移 / 幂等性 / 可观测性。

P0：至少快乐路径 + 一个边界 + 一个失败。P2：快乐路径 + 一个边界。
